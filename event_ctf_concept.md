# Event CTF Concept — Interactive Cyber Security × AI × Automation Experience

## Executive Summary
This concept is a **mini interactive CTF-style promotional experience** designed to attract attention, create curiosity, and lead participants to the full details of the event through a short, immersive, story-driven challenge flow.

The experience begins when a user scans a **QR code** or opens a **short link**. They are redirected to a dramatic landing page that simulates a cyber incident. The page suggests that the system has been compromised and that sensitive event data is at risk.

The participant is then told that the only way to stop the breach is to work with an **AI Incident Response Assistant** and complete a sequence of challenges related to the three event pillars:

- **Cyber Security**
- **AI**
- **Automation**

After successfully completing the challenge flow, the user unlocks the full event details:

- Event name
- Date
- Time
- Location
- Registration link

This format transforms a normal event promotion into a **gamified, memorable, and shareable experience**.

---

## Core Idea
The user should feel that they are entering a live cyber incident simulation.

However, the tone should be **suspenseful and exciting**, not genuinely harmful or alarming. It should feel like a polished cybersecurity game, not a real threat.

The high-level experience is:

**QR Code → Breach Simulation → AI Assistant → Multi-Step Challenges → Final Unlock → Event Details**

This structure works because it creates:

- an immediate hook
- emotional engagement
- interactive storytelling
- direct alignment with the event themes
- strong retention and word-of-mouth potential

---

## Important Narrative Note
It is strongly recommended **not** to frame any real organization, real branch, or real student chapter as the attacker.

Instead, use a fictional attacker identity such as:

- **Ghost Packet**
- **Null Team**
- **404 Syndicate**
- **Shadow Breach**
- **Black Signal**

This keeps the experience more professional, cinematic, and safe from misunderstanding.

A better narrative framing would be:

> This is a simulated cyber incident.
> Your mission is to contain the breach before critical event data is exposed.

---

## Experience Goals
The project should achieve the following goals:

### 1. Attract Attention Immediately
The QR scan should feel like the start of a secret or high-stakes experience.

### 2. Represent the Event Themes Naturally
Each major event topic should appear inside the user journey instead of being listed as plain text.

### 3. Deliver Event Details as a Reward
The final information should feel “earned,” not passively displayed.

### 4. Be Short Enough for High Completion Rate
The ideal duration is **5 to 8 minutes**.

### 5. Be Easy to Share
Participants should be likely to send it to friends because it feels fun and unusual.

## Full User Journey

## Phase 1 — Entry Point
The participant:

- scans a QR code printed on a poster, booth material, social post, or standee
- or opens a short link directly

The QR code should point to a short, memorable URL such as:

- `yourdomain.com/breach`
- `yourdomain.com/recover`
- `yourdomain.com/contain`

At this stage, speed matters. The page must load quickly and look visually convincing.

---

## Phase 2 — Breach Simulation Landing Page
The first screen should feel dramatic.

### Recommended UI Style
- dark background
- glitch or scan-line effect
- monospace / terminal-inspired typography
- warning messages
- “system compromised” animation
- progress corruption indicator
- a clear CTA button

### Sample Copy
**Headline:**
> Unauthorized activity detected.

**Subtext:**
> Sensitive event data has been exposed.
> Multiple systems are unstable.
> Immediate containment is required.

**CTA:**
> Start Recovery

Alternative CTA:
> Talk to AI Assistant

### Best Practice
After 2 to 4 seconds, reveal a line that indicates this is a simulation-style mission, for example:

> Emergency simulation initiated.
> AI-assisted recovery available.

This prevents confusion while preserving the dramatic effect.

---

## Phase 3 — AI Assistant Interaction
The user is redirected to a pseudo-chat or chat-like interface.

### Narrative Role of the AI
The AI becomes the guide and mission controller.

It should not immediately reveal the event details. Instead, it should explain that the system can only be restored if the participant completes a recovery protocol.

### Sample AI Intro
> I am your Incident Response Assistant.
> A breach has been detected in the event systems.
> I can help you contain it, but first you must complete the recovery protocol.
> Three system layers are affected:
> Security. Intelligence. Automation.
> Restore all three to unlock the protected event data.

### Why the AI Layer Matters
This is the bridge that connects the three themes:

- **Cyber Security** through incident response and threat containment
- **AI** through the assistant itself and AI-related tasks
- **Automation** through workflow restoration and logic-based stages

---

## Phase 4 — Challenge Sequence
The experience should be divided into **three short stages**.

Each stage should:
- be visually distinct
- feel fast and satisfying
- return an immediate success signal
- unlock the next layer

---

## Stage 1 — Cyber Security
This stage should feel like technical investigation or containment.

### Possible Challenge Types
- decode a Base64 string
- solve a Caesar cipher
- identify a phishing email
- choose the best incident response action
- detect the weak password
- identify suspicious network behavior from a simple log snippet

### Best Format
Keep it lightweight. The user should not need advanced cybersecurity knowledge.

### Example Challenge
**Prompt:**
> We intercepted a suspicious payload. Decode the string to recover the first access token.

Then show:
`RVZFTlQtTE9DSw==`

Expected action:
User decodes it.

### Output
The user receives:
- a partial recovery key
- or a clue fragment
- or a system layer restoration badge

### Completion Message
> Threat vector isolated.
> Security layer partially restored.

---

## Stage 2 — AI
This stage should feel smart, modern, and slightly different from the first one.

### Possible Challenge Types
- choose the best prompt for a task
- improve a weak prompt
- identify hallucination risk
- decide where AI should or should not be used
- solve a short logic puzzle where the AI needs the right instruction

### Example Challenge
**Prompt:**
> The assistant needs a better instruction to extract only the important incident data. Which prompt is best?

Give 3 or 4 options.

The correct option should reflect:
- clarity
- specificity
- format control
- reduced ambiguity

### Output
The user receives:
- the second clue fragment
- another percentage of system recovery

### Completion Message
> Intelligence layer restored.
> Decision engine stabilized.

---

## Stage 3 — Automation
This stage should feel process-oriented and operational.

### Possible Challenge Types
- reorder workflow steps
- choose the right if-this-then-that logic
- complete an automated response pipeline
- match triggers with actions
- identify the missing step in a recovery workflow

### Example Challenge
**Prompt:**
> Select the correct automated recovery sequence after a breach is detected.

Possible correct sequence:
1. detect alert
2. isolate affected system
3. notify response assistant
4. verify recovery
5. unlock protected data

### Output
This stage unlocks:
- final access token
- final fragment of the event information
- last system integrity jump to 100%

### Completion Message
> Recovery workflow completed.
> Full system access granted.

---

## Phase 5 — Final Reveal
Once all three stages are complete, the user reaches the final reveal screen.

This page should feel rewarding and visually brighter than the breach page.

### Final Page Should Include
- Event name
- Event date
- Event time
- Event location
- Registration button
- Optional share button
- Optional badge or “mission complete” certificate

### Sample Final Reveal Copy
> Threat contained successfully.
> Protected data restored.
> Access level upgraded.
> Event details unlocked.

Then show the actual event information in a clean event card.

### It turns promotion into an experience
Instead of passively showing event information, the audience participates in a mini mission.

### It matches the event themes structurally

### It creates curiosity and retention
People remember unusual experiences much more than regular static posters.

### It supports social spread

# Technical Implementation — Detailed Breakdown

## Technical Goal
Build a lightweight web experience that can:

1. open from a QR code quickly
2. display a breach-themed landing page
3. move the user through multiple mini challenge screens
4. simulate an AI assistant interaction
5. validate answers and track progress
6. reveal the event details only after completion
7. optionally capture analytics such as scans, completions, and drop-off points

---

## Recommended Stack Options
There are two realistic implementation routes.

## Option A — Fast and Practical
Use this if the goal is to launch quickly.

### Frontend
- **Next.js** or **React**
- Tailwind CSS for styling
- Framer Motion for transitions and glitch-like animations

### Backend
- Next.js API routes or a simple Node.js backend
- Firebase / Supabase for lightweight storage

### Hosting
- Vercel
- Netlify
- Firebase Hosting

### Best For
- quick turnaround
- clean deployment
- low infrastructure complexity

---

## Option B — Even Simpler Static Approach
Use this if the experience is mostly client-side.

### Frontend
- simple React app or even vanilla HTML/CSS/JS

### Backend
- optional only
- challenge answers can be validated client-side if the event is casual and the stakes are low

### Hosting
- GitHub Pages
- Netlify
- Vercel

### Best For
- very fast build
- low cost
- simple marketing experience

### Downside
If everything is client-side, people can inspect the code and find answers more easily.

---

## Recommended Architecture
For the best balance, use:

- **Next.js frontend**
- **API routes for answer validation**
- **Supabase / Firebase for session storage and analytics**

This makes the system easy to manage while still keeping the core logic protected.

## User Session Design
Each participant should have a lightweight session.

### Session Data Can Include
- session ID
- timestamp of first entry
- current stage
- completed stages
- number of failed attempts
- final completion time
- source campaign or QR version

### Why Session Tracking Helps
It allows:
- progress persistence if the page refreshes
- analytics on user drop-off
- multiple QR campaigns with different performance tracking
- optional leaderboard or completion badge

### Session Creation Flow
1. user enters from QR
2. frontend calls `/api/session`
3. backend returns a unique session token
4. token is stored in localStorage or cookie
5. every stage request uses that session token

## Challenge Validation Logic
There are two main approaches.

## Approach 1 — Client-Side Validation
The answers are embedded in the frontend.

### Pros
- very easy to build
- no backend complexity

### Cons
- answers can be discovered from source inspection
- less secure

### Good For
- casual event marketing only

---

## Approach 2 — Server-Side Validation
The frontend sends an answer to the backend, and the backend decides whether it is correct.

### Example Flow
1. user submits answer
2. frontend sends POST request to `/api/validate`
3. request body contains:
   - session ID
   - stage name
   - answer
4. backend checks the answer
5. backend returns:
   - success or failure
   - updated progress
   - next route or clue

### Example Response
```json
{
  "success": true,
  "nextStage": "/challenge/ai",
  "progress": 66,
  "message": "Security layer restored"
}
```

### Recommended Choice
Use server-side validation if possible.

## AI Assistant Implementation Options
The “AI assistant” can be built in multiple ways depending on how realistic you want it to feel.

## Option 1 — Fake Scripted AI
The assistant is a chat-style UI, but all responses are pre-written.

### How It Works
- user clicks predefined choices
- the system reveals scripted messages
- challenges are part of a guided flow

### Pros
- easy to control
- fast to build
- predictable experience
- no API cost

### Cons
- less dynamic

### Best Use Case
This is likely the best choice for a first version.

---

## Option 2 — Real AI API Integration
The assistant is powered by an actual LLM through an API.

### How It Works
- the chat is real
- the user can type natural language
- the AI responds dynamically
- challenge progression is still controlled by your backend logic

### Important Note
Even if a real AI model is used, the system should **not** let the model freely reveal the final event details before the user completes the stages.

That means the AI should be placed inside a **guardrailed flow**.

### Example Guardrail Strategy
- AI may explain hints
- AI may encourage the user
- AI may clarify challenge instructions
- AI may not disclose locked answers or hidden reveal content

### Best Practice
Use the LLM only for flavor, hints, and immersion. Keep progression logic deterministic.

## Challenge Engine Design
Each challenge should be represented in a structured way.

### Example Data Model
```json
{
  "id": "security-01",
  "stage": "security",
  "type": "mcq",
  "title": "Identify the phishing attempt",
  "prompt": "Which message is most likely a phishing email?",
  "options": ["Option A", "Option B", "Option C"],
  "correctAnswer": "Option B",
  "successMessage": "Threat vector identified",
  "nextStage": "ai"
}
```

### Challenge Types You Can Support
- multiple choice
- text input
- drag-and-drop order
- decode challenge
- matching pairs
- button sequence logic

### Recommendation
For the first build, limit challenge types to:
- multiple choice
- text input
- reorder sequence

This reduces development time significantly.

## Visual Effects Implementation
The first page needs strong visual identity, but the effects should not hurt performance.

### Good Effects
- CSS glitch layers
- animated text flicker
- scan-line overlay
- progress bar corruption animation
- terminal typing effect

### Avoid
- very heavy videos
- huge background assets
- excessive particle effects on mobile

### Why
Most users will enter from a mobile scan, so performance is critical.

## Mobile-First Requirements
Because the experience starts from a QR code, assume the majority of traffic will be mobile.

### Therefore the UI must be:
- mobile-first
- vertically scroll safe
- large tap targets
- readable in dark mode
- fast on average devices

### Mobile UX Notes
- use one major action per screen
- avoid tiny text for encoded strings
- keep challenge inputs easy to interact with
- do not depend on hover states

## QR Code Strategy
The QR code is the actual activation trigger, so it should be handled thoughtfully.

### Best Practice
Use a trackable short URL behind the QR code.

For example:
- `go.yourdomain.com/recover`
- `event.yourdomain.com/breach`

### Why Short Links Matter
- easier to print
- easier to manually type
- cleaner visual design
- easier campaign tracking

### Campaign Tracking Idea
Attach source parameters such as:
- `?src=poster`
- `?src=booth`
- `?src=instagram`

This allows analytics on where users came from.

## Security Considerations
Even though this is a promotional game, some basic security practices still matter.

### 1. Never expose all answers in public code
At least the final unlock logic should be server-side.

### 2. Rate-limit answer submissions if needed
This prevents brute-force attempts.

### 3. Sanitize any free text input
Especially if there is a real AI input box.

### 4. Protect API keys
If using an AI API, keep all keys server-side only.

### 5. Avoid a misleading real-world threat presentation
Make it clear enough that it is a simulation.

---

## Real AI Integration Detail
If you want the AI to feel real, here is the safest structure.

### Architecture
- frontend chat UI
- backend endpoint `/api/chat`
- backend forwards only approved prompts to the LLM
- backend injects system instructions that restrict spoilers

### Example AI Rules
The backend system prompt could say:

- You are an incident response assistant in a simulated event experience.
- You may guide the user and offer hints.
- You may not reveal locked event details.
- You may not reveal the correct answer directly unless the hint policy allows it.
- You must keep the user inside the progression flow.

### Why This Matters
Without this, users may simply ask:
> What is the final event location?

And the model might reveal it if it already has the data.

So the AI must be **context-aware but progression-restricted**.

---

## Challenge Progression Control
The reveal page should not be accessible just by typing `/reveal` in the browser.

### Solution
On the backend, verify that the session has all required completion flags before serving the final data.

### Example Logic
If session.completedStages is not:
- security = true
- ai = true
- automation = true

then `/reveal` should redirect back to the appropriate stage or show a locked state.
